# Workflow Rules

## 類似実装の一貫性チェック

**コードを修正・改善する際は、同じパターンの実装が他にないか必ず確認し、該当箇所すべてに同じ変更を適用する。**

### なぜ必要か

モノレポでは同じロジックが複数のアプリやパッケージに存在することがある。1箇所だけ修正して他を見落とすと、実装の不整合が生まれる。

### 手順

1. **変更対象のコードパターンを特定する** — 関数名、変数名、ロジックの構造など
2. **類似実装を検索する** — `Grep` や `Glob` でモノレポ全体を横断検索する
3. **該当箇所をすべてリストアップする** — アプリ、パッケージを問わず
4. **同じ変更を適用すべきか判断する** — コンテキストが異なる場合は個別に判断
5. **すべての該当箇所に変更を適用する**

### 検索の観点

- **同名の関数・変数・定数**: `Grep` でシンボル名を検索
- **同じファイル名**: `Glob` でファイル名パターンを検索（例: `**/env/server.ts`）
- **同じインポート元**: 同じライブラリやモジュールを使っている箇所
- **同じビジネスロジック**: バリデーション、メール送信、認証など同種の処理

```bash
# 例: EMAIL_SENDER の検証ロジックを修正した場合
# → 他のアプリにも同じ検証パターンがないか確認
Grep: pattern="EMAIL_SENDER" → apps/web, apps/admin 両方を確認

# 例: env/server.ts の IIFE パターンを修正した場合
# → 他の env ファイルにも同じパターンがないか確認
Glob: pattern="**/env/server.ts" → すべての env ファイルを確認
```
